## 01 · 관리자는 매일 반복되는 업무에 갇혀있습니다

배정, 계약, 체크인이 각각 다른 채널에 흩어지면서 운영자는 매번 직접 확인해야 했습니다.

- 연락 빈도가 아니라 상태 가시성이 진짜 문제였습니다.
- 운영자와 강사는 같은 수업을 서로 다른 정보 환경에서 경험했습니다.
- 프리랜서 83.7%가 관리자 커뮤니케이션 불량을 호소합니다. (Leapers 2024)
- 72%가 계약 취소·지연 스트레스를 경험합니다. (Leapers 2024)

**배정 불안 📞** 누가 갈 수 있는지 알 수가 없었습니다
후보는 많지만 상태가 보이지 않아 전화를 반복했습니다.

**계약 불안 📋** 서명이 됐는지 확인하려면 다시 물어야 했습니다
계약 상태가 실시간으로 보이지 않아 누락이 반복됐습니다.

**체크인 불안 🚗** 출발했는지 알 방법이 없었습니다
도착 시간을 모르니 대비도, 기록도 늦었습니다.

## 03 · **우리는 연락이 느린 게 문제가 아님을 발견했습니다**

운영자와 강사는 같은 수업을 완전히 다른 정보 환경에서 경험하고 있었습니다.

- 핵심 접점 3개 — 배정 요청, 계약 서명, 출강 체크인.
- 같은 상태를 다르게 해석하게 만드는 경험 구조가 문제였습니다.
- 30.2%가 서비스 품질 검증을 최우선 과제로 꼽습니다. (한국소비자원 2024)
- 13.3%만이 분쟁에 실제로 개입할 수 있었습니다. — 기록 자체가 없었기 때문입니다. (한국소비자원 2024)

> “필요한 것은 더 빠른 연락이 아니라, 확인하지 않아도 아는 상태였습니다.”

세 접점의 마찰을 따라가며 배정 지연의 진짜 원인을 다시 정의했습니다.

**3** 핵심 접점 — 배정 요청 · 계약 서명 · 출강 확인
**0** 확인 전화 — 상태가 보이면 물어볼 필요가 없습니다

---

### 표면 문제 vs 근본 문제

| 처음에 보인 문제                       | 진짜 문제                               |
| -------------------------------------- | --------------------------------------- |
| 카카오톡과 전화 중심이라 연락이 느리다 | 같은 상태를 같은 시간에 보지 못한다     |
| 계약서가 이메일로 따로 돌아다닌다      | 배정·계약·체크인이 연결되지 않는다      |
| 강사와 운영자가 다른 앱을 쓴다         | 운영 판단의 근거가 어디에도 남지 않는다 |

---

### 핵심 인사이트 3가지

**INSIGHT 1 — 속도가 아니라 가시성**
운영자가 원하는 것은 ’더 빠른 연락’이 아니라 ’확인하지 않아도 아는 상태’입니다. 연락을 더 많이 보내도 불안이 줄지 않았습니다.

**INSIGHT 2 — 세 단계는 하나의 여정**
배정·계약·체크인은 세 가지 기능이 아니라 하나의 연속된 여정입니다. 시스템이 이것을 분리하면 사람이 연결해야 합니다.

**INSIGHT 3 — 강사 UX가 운영 데이터를 만든다**
체크인 기록, 서명 완료, 응답 속도 — 이 데이터는 강사가 쉽게 행동해야 만들어집니다. 강사 앱의 UX가 나쁘면 운영 시스템의 데이터도 비어 있습니다.

## 04 · **배정, 계약, 체크인을 하나의 상태 흐름으로 묶었습니다**

개별 기능을 따로 자동화하는 대신, 상태 전이의 연결을 먼저 설계했습니다.

- 상태가 연결되면 운영자와 강사가 같은 지도를 보게 됩니다.
- AI는 이 흐름 위에서 추천과 알림을 제공합니다.

**State 01** 배정 확정
AI 추천으로 후보를 자동 정렬하고 추천 이유를 함께 보여줍니다.

**State 02** 계약 서명
전자서명과 확인 시점을 같은 흐름으로 실시간 동기화합니다.

**State 03** 출강 체크인
출발·도착·로그 기록을 운영 근거로 자동 남깁니다.

## 07 · Web · **배정에서 체크인까지, 운영자는 한 화면에서 모든 상태를 봅니다**

AI 추천으로 시작해 계약 확정, 출강 체크인까지 하나의 운영 지도가 완성됩니다.

- 배정→계약→체크인이 끊기지 않는 하나의 흐름으로 연결됩니다.
- 운영자는 판단하고, 시스템은 기록하고, AI는 브리핑합니다.

**Phase 01** AI 추천 & 배정 확정
미배정 수업에 AI 추천 강사 목록이 표시되고, 운영자가 최종 선택합니다.

**Phase 02** 계약 서명 & 확인
전자서명 요청 발송 → 강사 앱 서명 → 상태 자동 동기화.

**Phase 03** 출강 & 체크인 완료
GPS 알림 → 도착 체크인 → 운영 로그 & 정산 자동 기록.

## 09 · App · **서명 요청부터 완료까지, 재확인 없이 한 흐름으로 끝납니다**

운영자가 보내고, 강사가 앱에서 서명하면, 상태는 자동으로 동기화됩니다.

- 강사는 앱 안에서 계약서 확인과 전자서명을 모두 완료합니다.
- 서명 즉시 운영자 화면에 계약 확정 상태가 반영됩니다.

**Step 01** 서명 요청 발송
운영자가 앱에서 계약 내용 확인 후 서명 요청을 전송합니다.

**Step 02** 강사 앱 수신 & 서명
강사는 앱 안에서 계약서를 확인하고 전자서명을 완료합니다.

**Step 03** 상태 자동 동기화
서명 완료 즉시 운영자 화면에 계약 확정 상태가 반영됩니다.

## 10 · App · **체크인 한 번이 운영 기록과 정산 근거가 됩니다**

강사가 도착해 체크인하면 운영 로그와 정산 데이터가 동시에 생성됩니다.

출발 시간, 도착 시간, 체크인 시점이 운영 근거로 남습니다.

**Step 03** 정산 내역 자동 생성

## 11 · App **외부 계약서를 찍으면 앱이 알아서 인식하고 관리합니다**

수업 전 GPS 활성화 → 출발 감지 → 알림 발송 → 경로 기반 도착 예정 계산 → 도착·체크인 자동 기록

## 20 · AI 브리핑 · AI 기능 추가

**운영자가 아침에 앱을 열면 오늘 할 일이 먼저 나옵니다**

- 챗봇이 아닌 브리핑 형태 — MVP에서 신뢰도를 먼저 쌓는 방식입니다.

**AI 브리핑 흐름**
운영 이벤트 수집 → 서버 우선순위 판단 → LLM 자연어 요약 → 운영자 앱 브리핑 표시

**오늘 긴급** 미배정 수업 2건 — 내일 오전 수업, 아직 강사 미확정
**서명 대기** 계약 서명 3건 — 이달 내 완료 필요
**체크인 예정** 오후 2시 수업 강사 출발 미확인 — 1시간 30분 전

> “운영자는 판단하고, 시스템은 기록하고, AI는 브리핑합니다.”

### 14 · App 요약

**한 줄 메시지**
강사는 수락, 서명, 체크인만 하면 되고 나머지 기록은 시스템이 만든다.

- 강사 행동을 최소화해도 운영 데이터는 충분히 쌓인다.
- MLKit으로 외부 계약서도 앱 안으로 가져온다.
- GPS 알림으로 출발·도착 확인 전화를 없앤다.

**PPT 문장 예시**
강사에게 복잡한 운영 절차를 요구하지 않고, 필요한 행동만 순서대로 제시해 자연스럽게 기록이 남도록 설계했습니다.

- 배정 판단 시간 단축

- 커뮤니케이션 문제 감소 가능성

**PPT 문장 예시**
상태 연결 구조를 만들면 배정 속도, 입력 업무, 커뮤니케이션 비용이 동시에 개선됩니다.

**PPT 문장 예시**
